# 独自VM vs JSトランスパイル 比較

本プロジェクトでは Scratch 3.0 ベースの **AST ウォーキング型インタプリタ（独自VM）** を採用している。
このドキュメントでは、Phaser の JS にトランスパイルするアプローチとの違いを網羅的にまとめる。

## 1. デバッグ・トレーサビリティ

### VM（現在のアプローチ）

- ブロック単位で実行を追跡できる（DebugPanel のスレッドタブで「今どのブロックを実行中か」が見える）
- AST 構造をリアルタイムで可視化できる
- スプライト状態（x, y, direction, size 等）を 100ms 間隔で監視可能
- **ステップ実行**が原理的に実装しやすい（1ブロック実行 → 停止 → 再開）

### JSトランスパイル

- 実行が JS エンジンに委ねられるため、「今どのブロックを実行中か」の追跡が困難
- Source Map で対応は可能だが、ビジュアルブロックとの紐付けが複雑
- ブレークポイントはブラウザ DevTools に依存

## 2. リアルタイム実行・ホットリロード

### VM

- ブロックを編集 → 即座にスクリプト再ビルド → 実行中でも反映可能
- `buildScripts()` を毎秒再実行して AST 更新している
- **実行中にブロックを追加・削除しても安全**（次フレームから新しい AST で動く）

### JSトランスパイル

- コード変更のたびにトランスパイル → 再評価が必要
- 実行中の JS 関数を差し替えるのは非常に難しい
- 状態のリストア（座標、変数値）も別途必要

## 3. 実行制御（一時停止・速度調整・巻き戻し）

### VM

- `requestAnimationFrame` ループを止めるだけで一時停止
- `MAX_STEPS` を変えるだけで実行速度を調整可能
- フレームごとの状態スナップショットを取れば巻き戻しも実装可能
- 協調型マルチタスク（yield）で制御フローが明示的

### JSトランスパイル

- 一時停止は `debugger` 文の挿入や Generator 関数への変換が必要
- 速度調整は `setTimeout` / `requestAnimationFrame` の制御が複雑
- 巻き戻しは事実上不可能（JS の実行状態をシリアライズできない）

## 4. 並行実行（マルチスレッド）

### VM

- Scratch ライクなスレッドモデルが自然に実装できる
- 複数スプライトが同時にスクリプトを実行し、1フレーム内で公平にスケジューリング
- スレッドの状態（running / yield / promise_wait / done）が明示的に管理される

### JSトランスパイル

- JS はシングルスレッドなので、複数スプライトの並行実行を表現するには：
  - Generator 関数 + スケジューラを書く（→ 結局 VM に近づく）
  - Web Worker（→ DOM / Phaser との通信コストが大きい）
  - async/await で擬似的に（→ 制御が困難）

## 5. 安全性・サンドボックス

### VM

- ユーザーコードは `ScriptBlock` の opcode と args だけ。**任意の JS は実行されない**
- 無限ループ対策が組み込み済み（`MAX_STEPS=200` / フレーム）
- メモリ・CPU 使用量を制御可能

### JSトランスパイル

- 生成された JS が `eval` や `new Function` で実行される場合、XSS リスク
- 無限ループでブラウザがフリーズする可能性（Web Worker で分離しない限り）
- iframe sandbox で軽減可能だが、Phaser との連携が複雑になる

## 6. パフォーマンス

### VM（弱点）

- AST ウォーキングは JIT コンパイルされた JS より **10〜100倍遅い**
- ブロックごとに関数呼び出し + オブジェクトルックアップのオーバーヘッド
- ただし教育用途（数百ブロック規模）では体感差なし

### JSトランスパイル（強み）

- V8 / JavaScriptCore の JIT で最適化される
- 大量のスプライト・複雑な計算に強い
- プロダクションゲームに適している

## 7. エラーハンドリング

### VM

- ブロック単位で try-catch でき、「どのブロックでエラーが起きたか」を正確に特定
- エラー時にそのブロックをハイライト表示できる
- 不正な引数（文字列を数値に等）をブロック関数内で安全に処理

### JSトランスパイル

- スタックトレースからブロックへの逆引きが必要（Source Map）
- ランタイムエラーが Phaser の内部エラーと混在して分かりにくい

## 8. マルチプレイヤー同期（Colyseus との相性）

### VM

- 実行状態（スレッド、スプライト状態）がシリアライズしやすい構造
- サーバー側で同じ VM を動かして**権威サーバーモデル**が実装可能
- ブロック実行ログをそのまま同期データとして送れる

### JSトランスパイル

- JS 実行状態のシリアライズは困難
- サーバーでの実行には Node.js + Phaser headless が必要（重い）
- 状態同期は結果（座標等）のみになりがち

## まとめ

| 観点 | 独自VM | JSトランスパイル |
|------|--------|----------------|
| デバッグ性 | **◎** ブロック単位追跡 | △ Source Map 依存 |
| ホットリロード | **◎** 即座に反映 | △ 再コンパイル必要 |
| 実行制御 | **◎** 停止/速度/巻き戻し | × 困難 |
| 並行実行 | **◎** スレッドモデル内蔵 | △ Generator 等で擬似的に |
| 安全性 | **◎** サンドボックス | △ 追加対策必要 |
| パフォーマンス | △ 遅い | **◎** JIT 最適化 |
| エラー特定 | **◎** ブロック単位 | △ スタックトレース |
| マルチプレイ同期 | **◎** 状態シリアライズ容易 | × 困難 |
| 実装コスト | × 自前で全部書く | ◎ JS 生成するだけ |

教育用 VPL + マルチプレイヤーという要件では、VM アプローチが適している。パフォーマンスの弱点も、数百ブロック規模なら問題にならない。
