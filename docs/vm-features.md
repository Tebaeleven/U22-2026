# VMだからできる機能リスト

本プロジェクトの AST ウォーキング型インタプリタ（独自VM）だからこそ実現できる、
またはトランスパイルでは困難な機能を網羅的にまとめる。

## 1. 実行制御

| 機能 | 説明 | 実装難易度 |
|------|------|-----------|
| ステップ実行 | 1ブロックずつ実行→停止→再開。実行中ブロックをエディタ上でハイライト | 低 |
| スロー再生 | `MAX_STEPS` を下げるだけで実行速度を段階調整 | 低 |
| ターボモード | `MAX_STEPS` を上げて高速実行（Scratch互換） | 低 |
| 巻き戻し（タイムトラベル） | フレームごとにスプライト状態のスナップショットを保存→任意フレームに復元 | 中 |
| 実行の録画・リプレイ | 全フレームの状態変化をシリアライズして保存→再生 | 中 |

## 2. デバッグ・学習支援

| 機能 | 説明 | 実装難易度 |
|------|------|-----------|
| ブロック実行ヒートマップ | 各ブロックの実行回数をカウント→色の濃さで可視化 | 低 |
| 変数ウォッチ | 変数の値をリアルタイムでステージ上に表示（Scratch準拠） | 低 |
| ブレークポイント | 特定ブロックにマーカー→そこで実行停止。ブラウザDevTools不要 | 中 |
| 実行トレースログ | どのブロックがどの順で実行されたかをタイムライン表示 | 中 |
| 条件分岐の可視化 | if文の判定結果（true/false）をリアルタイムでブロック横に表示 | 低 |
| エラー箇所ハイライト | ゼロ除算やNaN発生時、原因ブロックを赤くハイライト | 低 |

## 3. ホットリロード・ライブコーディング

| 機能 | 説明 | 実装難易度 |
|------|------|-----------|
| 実行中ブロック編集 | ゲーム実行中にブロックの引数を変更→次フレームから即反映 | 低（既存） |
| 実行中スクリプト追加 | 走っている途中で新しいスクリプトチェーンを追加→即座に並行実行開始 | 低 |
| パラメータスライダー | 数値引数にスライダーUIを付けて、実行中にリアルタイム調整 | 中 |

## 4. マルチプレイヤー固有

| 機能 | 説明 | 実装難易度 |
|------|------|-----------|
| 権威サーバーVM | サーバー側で同じVMを動かし、クライアントは描画のみ。チート不可能 | 高 |
| ブロック実行の同期 | 「誰がどのブロックを今実行中か」を他プレイヤーにリアルタイム共有 | 中 |
| 共同編集+共同デバッグ | 他の人がステップ実行している様子をリアルタイムで見られる | 高 |
| 実行状態のスナップショット共有 | ゲーム状態をシリアライズして別プレイヤーに送信→同じ状態から再開 | 中 |
| サーバーサイドリプレイ判定 | 対戦結果をサーバーでリプレイして正当性を検証 | 中 |

## 5. 教育・分析

| 機能 | 説明 | 実装難易度 |
|------|------|-----------|
| 実行パス比較 | 「正解のプログラム」と「生徒のプログラム」の実行トレースを並べて差分表示 | 高 |
| ブロック使用統計 | どのブロックを何回使ったか、どの概念を理解しているか分析 | 低 |
| 無限ループ検出+説明 | `MAX_STEPS` に達した時「このループが終わらない理由」をブロック単位で説明 | 中 |
| 実行コスト可視化 | 各ブロック・ループの実行ステップ数をバーチャートで表示 | 中 |
| 段階的ヒント | 「次に実行されるブロック」を予測させるクイズモード | 中 |

## 6. 高度なVM制御

| 機能 | 説明 | 実装難易度 |
|------|------|-----------|
| カスタムスケジューラ | スレッドの優先度を変更。特定スプライトを他より多く実行させる | 中 |
| 条件付きyield | 「衝突したら一時停止」「特定変数が閾値を超えたら減速」等の動的制御 | 中 |
| スレッド間メッセージング | broadcast/receive でスプライト間通信。VM側で送受信を完全に管理 | 中 |
| スナップショット分岐 | ある時点の状態を保存→「もしここで別の選択をしたら？」をシミュレーション | 高 |

## 7. ゲームエンジン固有

| 機能 | 説明 | 実装難易度 |
|------|------|-----------|
| 衝突時コールバック（Hat） | `event_whentouching` をVMレベルで実装。毎フレーム衝突判定→スレッド起動 | 中 |
| 物理演算の決定的実行 | VM側で乱数シードを管理→同じ入力なら同じ結果を保証（リプレイ互換性） | 中 |
| スプライトクローン | Scratchのクローン機能。VMがスプライト+スレッドを動的に複製 | 中 |
| カスタムイベント | ユーザー定義のHatブロック（「HPが0になったとき」等） | 中 |

## VMならでは度ランキング

トランスパイルでは **原理的に困難** または **実装コストが桁違いに高くなる** ものを特にピックアップ：

1. **ステップ実行 + ブロックハイライト** — JS実行中に「今どのブロックか」を追跡するのは非常に困難
2. **タイムトラベルデバッグ** — JS の実行状態はシリアライズできない
3. **ブロック実行ヒートマップ** — JIT最適化されたJS内部の実行回数を取るのは不可能
4. **権威サーバーVM** — Node.js + Phaser headless はコストが大きすぎる
5. **実行パス比較** — 2つのJSプログラムの実行トレースを構造的に比較する手段がない
6. **決定的実行（リプレイ保証）** — JIT の浮動小数点最適化で結果が変わりうる
7. **実行中ブロック編集** — 実行中のJS関数を差し替えるのは極めて困難
